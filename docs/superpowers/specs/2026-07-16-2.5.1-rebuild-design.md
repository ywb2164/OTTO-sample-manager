# OTTO Sample Manager 2.5.1 重构建与覆盖安装设计

## 目标

发布一个可与旧 2.5.0 明确区分的 Windows x64 安装包，覆盖安装到 `E:\采样管理器`，并以安装后的真实二进制和真实桌面拖放链路作为验收对象。

## 已确认根因

用户当前安装的 `E:\采样管理器\otto-sample-manager.exe` 是 2026-07-16 11:04:52 构建的旧文件，SHA-256 为 `DA782368CA5D543DF3E9DA3C72B0CEDEB8EFED437E8F0F85BCBF9CF8C70949DB`，PE Subsystem 为 `3`（控制台）。仓库后来生成的 release 文件时间为 12:12:41，SHA-256 为 `40D1812331A294BE51A382F4280C2C9F51FF03D037CBDD102E2FEBF31B53AAA1`，PE Subsystem 为 `2`（Windows GUI）。相同的 2.5.0 版本号和安装包名称造成了旧产物混淆；拖放截图也来自旧二进制，尚未验证新 `CF_HDROP` 实现。

## 变更范围

1. 将 npm、Cargo 和 Tauri 应用版本统一提升为 `2.5.1`，同步锁文件。
2. 增加 Windows release 产物校验脚本，至少验证主程序 PE Subsystem 为 `2`、文件版本为 `2.5.1`、NSIS 文件名包含 `2.5.1`，并输出 SHA-256。
3. 在 CI 的 Tauri release build 后运行同一校验，避免控制台子系统或旧产物进入发布流程。
4. 清理本地旧的 2.5.1 构建目标后重新编译，避免复用旧应用二进制；签名私钥缺失时允许本地使用已成功生成的未签名测试安装包，但明确记录签名状态。
5. 静默覆盖安装到 `E:\采样管理器`，安装前不删除用户数据与 `Copy` 目录。
6. 安装后重新读取 EXE 的版本、SHA-256 和 PE Subsystem；必须与本次构建记录一致，且启动时不得创建 Windows Terminal/控制台宿主。
7. 使用安装后的真实应用从素材行发起拖拽，接收目标只要求能识别一个存在且非空的音频文件；不要求 Melodyne 或其他 DAW 验证。

## 拖放诊断门槛

先验证当前 `CF_HDROP` 实现在安装后的 2.5.1 完整链路中是否成功。若接收方仍不能识别文件，再记录从 React `dragstart`、Tauri `drag_start` 命令、SQLite 路径解析、副本准备到 `SHDoDragDrop` 返回值的边界信息，并只修复被证据确认失败的层。不会在未复现前继续叠加 OLE 实现改动。

## 测试与验收

- `npm test`、`npm run check:types`、`cargo test --all-targets`、严格 Clippy 和格式检查通过。
- 2.5.1 release 主程序的 PE Subsystem 等于 `2`。
- NSIS 安装包名称和应用版本均为 2.5.1。
- 覆盖安装后的 EXE 哈希等于本次 release EXE 哈希，Subsystem 仍为 `2`。
- 启动安装后的应用时没有附带终端。
- 从安装后的应用拖出音频时，接收方检测到存在且非空的文件。

## 非目标

- 不删除或重建用户素材数据库。
- 不验证 DAW 特定导入语义。
- 不在仓库中保存 Tauri 更新签名私钥。
- 不顺带修改无关 UI 或库结构。
